# GlassGPT 5.4.0 Release Design

> User brief treated as the approved product spec for this session so implementation can continue without an extra review stop.

**Goal:** Ship GlassGPT 5.4.0 on `codex/stable-5.4` with working image upload, file upload, polling-friendly tool call indicators, sandbox file download/preview, and a full backend+iOS release.

**Architecture:** The 5.4.0 release keeps the post-5.3 backend-owned architecture intact: iOS sends conversation messages to the Cloudflare Workers backend, the backend calls the OpenAI Responses API, persists assistant state into D1, and iOS polls conversation detail every 250 ms during active runs. The design restores 4.12.6 feature parity without reintroducing on-device OpenAI access or switching back to SSE.

**Constraints:**
- Do not merge or port code from `stable-4.12`; use it only as UX reference.
- Do not change the polling architecture documented in `docs/audit-5.3.2.md`.
- Do not use skip flags in CI, deploy, or TestFlight release scripts.
- Keep the OpenAI API key backend-only.

## Investigated Root Causes

### Feature 1: Image Upload

Confirmed code inspection:
- The request DTOs, workflow params, execution request, and OpenAI adapter already thread `imageBase64`.
- `buildInputMessages()` in `services/backend/src/adapters/openai/openai-responses.ts` already emits `input_image` parts.

Confirmed regression:
- `services/backend/src/http/app.ts` applies `bodyLimit({ maxSize: 1024 * 1024 })` to `*`, which also affects `POST /v1/conversations/:conversationId/messages`.
- A base64-encoded JPEG can easily exceed 1 MB even when the original image is modest, so the backend rejects or truncates the request before the image ever reaches the route handler.

Design decision:
- Add a larger body limit for message-creation requests while keeping the small default limit for unrelated JSON endpoints.
- Add explicit regression tests around large message bodies and image-bearing OpenAI request construction.

### Feature 2: File Upload

Confirmed code inspection:
- `BackendConversationProjectionController+Actions.swift` already uploads pending files before calling `sendMessage`.
- `services/backend/src/http/routes/files.ts` already proxies multipart uploads to OpenAI and returns `{ fileId, filename, bytes }`.
- `buildInputMessages()` already emits `input_file` parts for `fileIds`.

Confirmed regressions:
- The same global 1 MB `bodyLimit` in `services/backend/src/http/app.ts` also applies before `/v1/files/upload`, so large file uploads are still capped by the global middleware despite the route-specific 50 MB middleware.
- `MessageInputBar` allows attachment-only sends, but `BackendConversationProjectionController.sendMessage(text:)` rejects empty trimmed text, and `createMessageRequestSchema` requires `content.min(1)`.
- `buildInputMessages()` always emits an `input_text` part, even when the message body is empty and only attachments should be sent.

Design decision:
- Make attachment-bearing messages valid even when the user text is empty.
- Allow the backend request schema and OpenAI adapter to encode attachment-only user messages without an empty text part.
- Add regression tests for upload lifecycle, request payload shape, and attachment-only sends.

### Feature 3: Tool Call Indicators

Confirmed code inspection:
- `chat-run-execution-operations.ts` persists `toolCallsJSON` on every `tool_call_updated` event via `persistAssistantSnapshot()`.
- `dto-mappers.ts` attempts to map `toolCallsJSON` into `MessageDTO.toolCalls`.
- `ConversationSyncApplication/BackendProjectionStore+ProjectionMapping.swift` maps `ToolCallInfoDTO` into `ToolCallInfo`.
- The polling loop in `BackendConversationRunStreamDriver.swift` calls `applyLiveOverlayFromPolledMessages()` on every iteration.

Confirmed regressions:
- `live-payload-codec.ts` persists optional tool-call properties as explicit `null`, while the contract schema used by `dto-mappers.ts` expects those optional properties to be omitted instead of `null`.
- As a result, persisted `toolCallsJSON` can be dropped during conversation-detail DTO mapping even though the backend correctly wrote it during the active run.
- The current grace-period override in `BackendConversationProjectionController+StreamLifecycle.swift` only rewrites completed tool calls while polling is active.
- When a tool call is first observed in completed state near the end of a run, the same poll iteration can immediately see the run become terminal, finalize the conversation, and call `clearLiveSurface()`, removing the live overlay before the indicator remains visible for the intended grace period.

Design decision:
- Keep the backend persistence model, but normalize stored tool-call JSON so conversation-detail DTOs always surface persisted tool calls.
- Update the client polling overlay to anchor on the active run's assistant message and retain grace-period indicators across finalization briefly before pruning them by age.
- Add tests for backend codec/DTO normalization, pre-completion tool-call persistence, and client grace-period visibility.

### Feature 4: Sandbox File Download & Preview

Confirmed code inspection:
- `services/backend/src/http/routes/files.ts` already attempts both container-scoped and direct file download endpoints and streams the response body back.
- iOS already contains generated-file infrastructure:
  - `GeneratedFilesCore/GeneratedFilePolicy.swift`
  - `GeneratedFilesCache/*`
  - `ChatPresentation/FilePreviewStore.swift`
  - `NativeChatUI/Components/FilePreviewSheet.swift`

Confirmed regressions:
- `BackendClient+FileRequests.swift` builds download URLs with `appendingPathComponent()` after embedding `?container_id=...` in the path string, which percent-encodes the query separator and breaks container-scoped downloads.
- `MessageBubble` supports `onSandboxLinkTap`, but the backend chat/agent message list never passes a callback, so sandbox links do nothing.
- `FilePreviewStore` exists but is not wired into the shell state or root view, so there is no live presentation path for preview/share state.
- There is no coordinator that matches tapped links to `FilePathAnnotation`, checks the generated-file cache, downloads through the backend client, stores the file, and presents either preview or share UI.

Design decision:
- Fix backend client URL construction with `URLComponents`/`makeURL`.
- Add a generated-file interaction coordinator that reuses `GeneratedFileAnnotationMatcher`, `GeneratedFilePolicy`, `GeneratedFileCacheManager`, and `FilePreviewStore`.
- Wire preview/share presentation into `NativeChatShellState` and `ContentView` using existing `FilePreviewSheet` and share-sheet support.
- Add tests for URL construction, annotation matching, download->cache->presentation flow, and callback wiring.

### Versioning and Release

Confirmed release surfaces:
- Backend versions are hard-coded in package metadata, connection-check logic, deploy smoke-check defaults, and several backend/iOS tests.
- iOS fallback/backend version defaults are hard-coded in multiple Swift sources and tests.

Design decision:
- Update every required 5.3.2/5.3.x hard-coded version surface to 5.4.0/20223.
- Run both CI scripts cleanly, then production deploy and TestFlight release scripts with no skip flags other than the allowed `--skip-main-promotion`.

## Chosen Implementation Approach

1. Fix attachment transport first because image/file regressions share request-size and message-shape issues.
2. Stabilize tool-call visibility entirely on the client polling path unless investigation during tests proves backend persistence is insufficient.
3. Reuse existing generated-file cache and preview primitives rather than building new UI or storage layers.
4. Treat each feature as TDD-backed, but verify the integrated flows with full backend/iOS CI before release.

## Verification Strategy

- Backend unit/integration tests for:
  - large message body acceptance on the message route
  - upload proxy request/response behavior
  - OpenAI request payload shape for `input_image` and `input_file`
  - persisted tool-call live state during active runs
  - tool-call DTO normalization from persisted JSON into conversation detail
  - sandbox file proxy response behavior
- iOS tests for:
  - attachment-aware send acceptance and upload lifecycle
  - request payload construction
  - tool-call grace-period retention
  - generated-file download URL construction
  - sandbox link callback wiring and preview/share state updates
- Final proof:
  - `./scripts/ci_backend.sh`
  - `./scripts/ci_ios.sh`
  - `./scripts/deploy_backend.sh --env production`
  - `./scripts/release_testflight.sh 5.4.0 20223 --branch codex/stable-5.4 --skip-main-promotion`
